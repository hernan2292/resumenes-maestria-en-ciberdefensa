El Problema de Diseño

Confidencialidad vs. Capacidad de Búsqueda

Diapo 2/9

En la criptografía simétrica clásica, la seguridad exige indistinguibilidad:

  • Cifrado Probabilístico (IND-CPA): Un mismo texto plano genera textos cifrados distintos cada vez (usando un IV/nonce único en AES-GCM o AES-CBC).
  • Destrucción de la Estructura: El texto cifrado actúa como una caja negra probabilística. El servidor no puede indexar ni filtrar.
  • Incompatibilidad Operativa: Buscar una palabra clave obliga al cliente a descargar todos los datos cifrados y descifrarlos localmente.
  • Cifrado Determinista (Falla de Diseño): Mismo texto plano siempre genera mismo texto cifrado. Altamente vulnerable a ataques de análisis de frecuencia.
Palabra:
Tensión clásica: A mayor indistinguibilidad matemática, menor capacidad de cómputo delegado.
Preliminares Académicos

El Núcleo Criptográfico Simétrico

Diapo 3/9

Las construcciones eficaces de SSE se asientan exclusivamente sobre primitivas simétricas convencionales:

  • Funciones Pseudoaleatorias (PRFs): Familias de funciones que mapean claves y entradas a salidas deterministas pero indistinguibles de ruido aleatorio para agentes externos.
    F: {0,1}λ × {0,1}m → {0,1}n
  • Instanciación Clave: HMAC-SHA256
    HMACK(w) = H((K ⊕ opad) || H((K ⊕ ipad) || w))
  • Esquema de Cifrado Probabilístico (SKE): Para resguardar los archivos, se utiliza AES-256 en modo GCM, garantizando confidencialidad **IND-CPA** e integridad (AEAD).

Ciclo de Vida de los Datos

Los Cuatro Algoritmos de SSE

Diapo 4/9

Un esquema SSE opera bajo un modelo de Servidor Semi-Honesto (Honest-but-Curious) mediante cuatro etapas:

  1. Setup(1λ): El cliente genera localmente su conjunto de claves simétricas secretas (K1, K2, K3).
  2. BuildIndex(K, D): El cliente extrae las palabras, genera un índice invertido estructurado y lo cifra junto con los archivos. Sube el índice cifrado (I) y los archivos cifrados (C) al servidor. Borra los planos.
  3. Trapdoor(K, w): Para buscar, el cliente genera un token simétrico determinista Tw para la palabra w, sin revelar su valor plano.
  4. Search(I, Tw): El servidor usa el Trapdoor para evaluar el índice a ciegas y retorna el conjunto de identificadores coincidentes.

Evolución Histórica

Esquema Secuencial de Song, Wagner y Perrig

Diapo 5/9

El nacimiento del cifrado buscable se basó en cifradores de flujo simétricos y PRFs:

  • Construcción: Cada palabra Wi se cifra individualmente. Se genera un valor pseudoaleatorio Xi = (Li, Ri) y una constante de comprobación Si = Fki(Li). El ciphertext final es Ci = Wi ⊕ (Li || Si).
  • Búsqueda Secuencial Lineal: Para buscar, el cliente debe enviar partes de las claves de derivación. El servidor debe escanear linealmente cada palabra de la base de datos completa.
  • Desventaja Principal: Complejidad de búsqueda lineal O(N × L) en el servidor, inviable para Big Data, y revelación de la ubicación exacta de palabras.

La Búsqueda Sublineal mediante Índices Invertidos

Diapo 6/9

Curtmola, Garay, Ostrovsky y Yong introdujeron la estructura de **Índice Invertido Cifrado (SSE-1)**, logrando búsquedas en tiempo óptimo **O(t)**:

  • Listas Enlazadas Cifradas: Cada palabra clave posee una lista enlazada donde cada nodo almacena cifrado el ID del documento y la dirección de memoria del siguiente nodo:
    Vj = EncKw(idj || Addrj+1)
  • Direccionamiento Ciego: Los nodos se almacenan de forma desordenada en una tabla hash en direcciones pseudoaleatorias Addrj = PRFK1(w || j).
  • Trapdoor Dual: El cliente envía Tw = (T1(w), Kw). El servidor localiza el primer nodo en T1, descifra con Kw y salta a la dirección del siguiente nodo secuencialmente.

Criptoanálisis de Datos Cifrados

El Peligro del Leakage (Goteo Criptográfico)

Diapo 7/9

Para lograr búsquedas sublineales sin descifrar, los esquemas de SSE toleran goteos de datos estructurados:

  • Search Pattern (Patrón de Búsqueda): Revela si el cliente busca la misma palabra clave. El servidor aprende si Tw,i == Tw,j.
  • Access Pattern (Patrón de Acceso): Revela qué documentos exactos coinciden con las consultas. El servidor aprende la lista de IDs devueltos.
  • Ataques Criptoanalíticos:
    • Inyección de Archivos (Zhang et al.): El servidor inyecta correos o logs diseñados de forma binaria. Al ver qué archivos se acceden, recupera el 100% de los Trapdoors.
    • Ataques de Volumen: Si el servidor tiene información estadística auxiliar, cruza las frecuencias de palabras reales con los tamaños de resultados devueltos.

Técnicas de Robustecimiento

Mitigación de Fugas en el Dominio Simétrico

Diapo 8/9

El cliente puede neutralizar de forma autónoma el análisis de fugas en el servidor mediante técnicas de enmascaramiento:

  • Padding de Documentos y Volumen: Cifrar archivos a tamaño fijo (ej. PKCS#7) y añadir IDs de documentos ficticios (Dummies) a las listas enlazadas para que todas las palabras parezcan tener igual cantidad de resultados.
  • Consultas Dummy (Ruido): Generar búsquedas periódicas ficticias de forma automatizada por el cliente para aplanar la frecuencia del Search Pattern.
  • ORAM (Oblivious RAM): Barajar y recifrar la memoria del servidor continuamente tras cada acceso para eliminar el Access Pattern (aumenta el coste de red).
  • Forward & Backward Privacy: Asegurar que nuevas inserciones de archivos no se correlacionen con búsquedas pasadas usando claves simétricas evolutivas.

Cierre de la Exposición

Conclusiones y Caso de Uso en Ciberdefensa

Diapo 9/9

El Cifrado Simétrico Buscable (SSE) es un puente práctico entre la confidencialidad absoluta de los datos y el procesamiento eficiente en la nube:

  • Eficiencia Práctica: Evita la altísima sobrecarga computacional de tecnologías en desarrollo como el cifrado homomórfico completo (FHE).
  • Caso SOC en Entornos Zero Trust: Permite almacenar logs y syslogs confidenciales en almacenamiento distribuido de terceros de forma inmutable.
  • Preservación de la Privacidad del Analista: El analista del SOC realiza búsquedas de indicadores de compromiso (IoC) sin revelar a los nodos distribuidores qué amenaza está investigando.
  • El Desafío Actual: La mitigación activa del leakage por parte del cliente es vital para evitar el perfilado pasivo del servidor.

Discurso Paso a Paso

Lo que debes decir exactamente:

"Para entender la necesidad de SSE, debemos revisar el conflicto fundamental de la criptografía simétrica clásica. Si tomamos un algoritmo estándar como AES-GCM en un esquema SKE, la seguridad exige que el cifrado sea probabilístico. Esto significa que si ciframos la misma IP maliciosa o el término 'failed' dos veces, obtendremos dos textos cifrados completamente distintos. Esto se conoce formalmente como seguridad IND-CPA."

"Esta indistinguibilidad destruye cualquier estructura u orden, convirtiendo los datos en una caja negra impenetrable. Para que un servidor haga una búsqueda tradicional, nos vemos forzados a descargar toda la base de datos cifrada, consumiendo ancho de banda, y descifrarla localmente. Esto no escala.

Por otro lado, si usáramos cifrado determinista ingenuo para habilitar la búsqueda directa, perderíamos la seguridad semántica, ya que el servidor deduciría los contenidos por análisis de frecuencia de palabras. SSE surge para resolver este dilema de diseño de forma óptima."

"Un esquema formal de Cifrado Simétrico Buscable se define a través de un protocolo interactivo cliente-servidor de cuatro algoritmos básicos."

"Primero, en la fase de Setup, el cliente inicializa sus claves secretas en su entorno local y seguro. Discurso Paso a Paso

Lo que debes decir exactamente:

"La verdadera revolución práctica de SSE ocurrió en 2006 con el esquema propuesto por Curtmola, Garay, Ostrovsky y Yong. Ellos cambiaron radicalmente el diseño pasando de un escaneo secuencial a una estructura de índice invertido cifrado basada en listas enlazadas en una tabla hash desordenada."

"En esta construcción, a cada palabra del diccionario se le asocia una lista enlazada de identificadores de los documentos que la contienen. El cliente cifra de manera independiente cada nodo de la lista usando una clave temporal derivada de la palabra, ocultando el ID del documento y la dirección física del siguiente nodo de la cadena en el servidor. La dirección de almacenamiento físico de cada nodo en el servidor es pseudoaleatoria."

"Cuando el analista busca, envía un Trapdoor dual compuesto por la dirección del primer nodo y la clave temporal del enlace. El servidor puede saltar directamente a la dirección inicial, descifrar a ciegas el primer nodo para obtener el ID de log y la dirección del segundo nodo, y continuar la navegación. La complejidad de la búsqueda pasa a ser estrictamente sublineal, tardando un tiempo proporcional únicamente al número de coincidencias reales, sin importar el tamaño total de la base de datos."
